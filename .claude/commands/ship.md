# /ship — 커밋부터 배포까지 한 번에

미커밋된 변경분을 커밋하고, 푸시하고, main에 머지해서 Vercel 프로덕션 배포까지 트리거한다.

## 실행 순서

1. **현재 상태 확인**
   - `git status --short` 와 `git branch --show-current` 로 변경 내역과 현재 브랜치를 파악한다.
   - 변경 내역이 전혀 없고 푸시할 커밋도 없으면 "배포할 변경분이 없습니다"라고 알리고 중단한다.

2. **빌드 검증 (프로덕션 안전장치)**
   - 프로덕션에 배포되므로 먼저 `yarn build` 를 실행해 타입/빌드 에러가 없는지 확인한다.
   - 빌드가 실패하면 **여기서 중단**하고 에러를 사용자에게 그대로 보여준다. (커밋/푸시/머지 금지)

3. **커밋**
   - `$ARGUMENTS` 로 커밋 메시지가 주어지면 그대로 사용한다.
   - 메시지가 없으면 `git diff` 내용을 보고 변경 성격을 요약한 **한국어 커밋 메시지**를 직접 만든다. (이 저장소의 기존 커밋 스타일에 맞춤)
   - `git add -A` 후 커밋한다. 커밋 메시지 끝에는 아래 트레일러를 붙인다:
     ```
     Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>
     ```

4. **푸시 + 머지 (브랜치에 따라 분기)**
   - **현재 브랜치가 `main`이면**: 머지 단계를 생략하고 `git push origin main` 만 한다.
   - **현재 브랜치가 작업 브랜치(main이 아니면)**:
     1. `git push origin <현재브랜치>`
     2. `git checkout main`
     3. `git pull origin main` (최신화)
     4. `git merge --no-ff <작업브랜치>`
     5. `git push origin main`

5. **배포**
   - main에 푸시되면 Vercel(GitHub 연동)이 자동으로 빌드·프로덕션 배포한다. **별도 명령은 실행하지 않는다.**
   - 사용자에게 "main 푸시 완료 → Vercel 자동 배포가 트리거되었습니다"라고 알리고, Vercel 대시보드에서 배포 진행 상황을 확인할 수 있음을 안내한다.

## 안전 규칙
- 빌드 실패 시 절대 커밋/푸시/머지를 진행하지 않는다.
- 머지 충돌이 발생하면 멈추고 사용자에게 충돌 파일을 알린다 (임의로 해결하지 않는다).
- 강제 푸시(`--force`)는 사용하지 않는다.
